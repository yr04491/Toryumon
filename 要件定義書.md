### コンセプト
「先生の板書の間違いを見つけて指摘する」という**ざまぁ感**を学習動機に変換する教育Webアプリ。
早押し×コメディ演出で正しい知識を記憶に強く定着させる。

### 学習効果の根拠
- **サプライズ効果**：信頼できそうな先生が間違えることで予測誤差が生まれ、記憶への定着が強くなる
- **ハイパーコレクション効果（高確信度エラー効果）**：自信満々な間違い→訂正という流れは通常の暗記より学習効果が高い

### 独自性
既存の「間違い探し」は視覚パズル（左右の絵の差）がほぼ全て。
**「板書の正誤を知識ベースで早押し判断する」学習アプリは類似例なし。**
また、先生のセリフではなく板書を疑う構造はメタ認知・批判的思考を促す点でも従来と異なる。

## 3. 登場キャラクター

### 先生
- 教科ごとに別キャラクター
- 見た目・口調は「ちゃんとした先生」
- 自信満々に説明し、たまに間違える（本人は気づいていない／見栄を張っている）
- 誤指摘を受けた場合：得意げにユーザーを馬鹿にする
- たまに「次は気をつけるよ」と成長する描写を挟む（いじめ感の緩和）

### 校長
- 全教科共通の1キャラクター
- 昔ながらの厳格なガミガミ校長（ゆるいノリではなく、本気で怒る昔気質）
- 正しい指摘を受けると怒りに飛んできて先生をガミガミ叱る

### 山田くん（ライバル）
- 全教科共通のクラスメートキャラ
- 普通〜やや出来る子くらいの立ち位置

| 間違い発生からの経過 | 山田くんの状態 |
|---------------------|----------------|
| 0〜3秒 | 何も起きない |
| 3〜7秒 | そわそわアニメーション（間違い存在のヒント） |
| 7〜10秒 | 手を挙げて先に指摘してしまう |

### 子供（ユーザー）
- プレイヤー本人
- 板書の間違い箇所をタップして指摘する
- 「なぜ間違いか」を選択肢で回答する

## 5. ゲームメカニクス詳細

### 授業フェーズ
- 先生が板書しながら授業を進行する
- 板書内容（数式・漢字・年号・図表など）は少しずつ追加される
- **間違いは板書（黒板の表示内容）のみ。先生のセリフは対象外。**

### 間違い発生
- 板書内容にランダムで間違いが混ざる
- **間違いが発生しないステージも存在する**（正しくスルーする判断も問われる）

### 指摘フェーズ
- 制限時間バーなし
- ユーザーは自分のペースで板書を観察する
- 間違いを発見したら該当箇所をタップして指摘
- 山田くんの動作がタイミングの目安（上記テーブル参照）

### 結果フェーズ（5パターン分岐）

| 状況 | 結果 |
|------|------|
| 正しく指摘・山田くんより早い（7秒以内） | 校長登場・先生をガミガミ説教 → 理由説明へ → スコア加算 |
| 山田くんに先を越された（7秒超） | 山田くんが指摘、横で悔しく見るだけ → 山田くんにスコア加算 |
| 間違いなしなのにタップ（誤報告） | 先生に馬鹿にされる → スコアペナルティ |
| 間違いあり・スルー（山田くんに越された） | 山田くんが指摘、見逃した悔しさを演出 |
| 間違いなし・正しくスルー | 授業終了「よく見てたね」演出 → スコア加算 |

### 校長フェーズ（正しく指摘した場合のみ）
- 校長が怒りに飛んでくる演出（ゲーム画面上にポップアップ）
- 先生をガミガミ叱る
- 先生のリアクション（汗、しょげる等）はバリエーション・コレクション要素

### 理由説明（学習の核）
- 「なぜ間違いか」を選択肢でタップ回答
- 正答でスコア加算 + 解説テキストを表示

### スコアシステム
- ユーザーと山田くんの正解指摘数を対戦形式で競う
- 各ステージ終了時に「ユーザー X回 vs 山田くん Y回」を表示
- 山田くんより多く正解すれば勝利

## 7. 技術スタック

| 層 | 採用技術 | 理由 |
|----|----------|------|
| 言語 | TypeScript | 型安全。JSONデータの型定義が楽 |
| ビルド | Vite | 設定ゼロで即起動、`npm run build` で純粋な静的SPAを出力 |
| UIフレームワーク | React | 状態→UI更新がリアクティブに書ける。「状態に応じた画像差し替え」と相性良好 |
| スタイル | CSS Modules（or Tailwind） | コンポーネント単位でスコープされたスタイル |
| コンテンツ管理 | JSON（静的ファイル） | ステージデータを外部ファイルで管理。フォーマットを統一すれば量産が容易 |
| デプロイ | GitHub Pages / Netlify（静的SPA） | `npm run build` の出力をそのままホスティング可能 |

### 採用しない技術・理由

| 技術 | 採用しない理由 |
|------|---------------|
| Next.js | SSR・SEO・APIルートが全て不要。静的配信が複雑になる。コンテスト提出には Vite の方がシンプル |
| Phaser / Unity | Canvas/ゲームエンジンはテキスト主体・UI主体の今回の設計に逆に手間 |
| Vanilla TS（FWなし） | 状態→UI配線を5画面分自前管理することになりバグのリスクが高い |

## 8. UIデザイン方針

### 参考イメージ
「ママにゲームを隠された」のような**ローファイ・手書き風・シンプル**なデザイン。
「できるだけイラストと動きだけで表現する」「説明なしで直感的に遊べる」を目標とする。

### 基本方針

| 項目 | 方針 |
|------|------|
| アニメーション | **なし**。CSS transition / keyframes は使わない |
| 動き・演出 | 画像の**切り替え**（src差し替え）と**移動**（position変更）のみで表現 |
| ビジュアル | 手書き風・シンプルなイラスト |
| フォント | 手書き・黒板風フォント（例：Google Fonts Noto Sans JP） |
| カラー | 黒板のグリーン・チョーク白を基調 |
| UIパーツ | グラデーション・シャドウなし。フラットでシンプル |

### 画像管理・演出の実装方針
- キャラクターの状態ごとに画像ファイルを用意（例：`teacher_normal.png` / `teacher_sweat.png`）
- React の state に応じて `<img src={...}>` を差し替えることで状態を表現
- 校長の登場は `position` をステップ変更（CSS transition なし）
- **ゲーム開始前に必要な画像を全てプリロード**し、差し替え時の遅延をゼロにする

```ts
// ステージ開始前に実行するプリロード
function preloadImages(urls: string[]): Promise<void[]> {
  return Promise.all(
    urls.map(url => new Promise<void>((resolve) => {
      const img = new Image();
      img.onload = () => resolve();
      img.src = url;
    }))
  );
}
```

## 9. コンテンツデータ構造（JSONスキーマ）

ステージデータをJSONで管理し、フォーマットを統一することでコンテンツ量産を容易にする。

### ステージ定義

```json
{
  "stageId": "elementary_math_grade3_stage01",
  "school": "elementary",
  "grade": 3,
  "subject": "math",
  "teacherId": "teacher_math",
  "blackboard": [
    {
      "id": "item_01",
      "type": "text",
      "content": "3 × 4 = 12",
      "isCorrect": true
    },
    {
      "id": "item_02",
      "type": "text",
      "content": "5 × 6 = 35",
      "isCorrect": false,
      "correctContent": "5 × 6 = 30",
      "question": {
        "choices": [
          { "text": "5×6は30が正しい", "isCorrect": true },
          { "text": "5×6は25が正しい", "isCorrect": false },
          { "text": "5×6は40が正しい", "isCorrect": false }
        ],
        "explanation": "5×6は、5を6回足した数なので30になります。"
      }
    }
  ],
  "yamada": {
    "worriedDelay": 5,
    "answerDelay": 9
  }
}
```

### キャラクター定義

```json
{
  "teacherId": "teacher_math",
  "name": "数学の先生",
  "images": {
    "normal": "/assets/characters/teacher_math_normal.png",
    "confident": "/assets/characters/teacher_math_confident.png",
    "sweat": "/assets/characters/teacher_math_sweat.png",
    "smug": "/assets/characters/teacher_math_smug.png",
    "growth": "/assets/characters/teacher_math_growth.png"
  }
}
```
